# Impeccable Design Context

## Design Context

### Users

**Primary Audience**: Dual-purpose platform serving both technical peers (developers, designers) and potential clients seeking to hire for projects or services.

**Context**: Users arrive looking to:
- Evaluate technical capabilities and systematic thinking (peers assessing skill/craft)
- Discover tools and services built by Max
- Assess fit for hiring/collaboration (clients)
- Explore the design system and component library as reference/inspiration

**Mindset**: These are people who appreciate meticulous craft, systematic approaches, and technical depth. They value substance over flash, precision over decoration.

### Brand Personality

**Three Words**: Retro, Swiss design, Performant

**Emotional Goals**: 
- **Nostalgic delight** - Warm feelings about early computing, terminal aesthetics, and the era when constraints bred creativity
- **Confidence in precision** - Everything feels intentional, systematically considered, and purposefully executed
- **Curiosity and exploration** - Encourages poking around to discover the systematic thinking behind the interface

**Voice & Tone**:
- Technical but not cold - there's warmth in the retro aesthetic
- Precise and systematic - Swiss design rigor applied to modern web
- Quietly confident - quality speaks for itself without needing to shout
- Performance-minded - fast, efficient, no bloat

### Aesthetic Direction

**Visual Direction**: Technical brutalism with retro computing nostalgia

**Core Visual Language**:
- **Terminal-inspired foundation**: Monospaced typography (IBM Plex Mono, Departure Mono), phosphor green accents reminiscent of CRT terminals, command-line interface patterns
- **Swiss design rigor**: Mathematical spacing systems, grid-based layouts with intentional breaks, systematic token architecture, minimal decoration
- **Zero-tolerance minimalism**: No border radius, no gradients, no soft shadows - only what's essential
- **Retro computing aesthetic**: Pixelated bitmap fonts, ASCII patterns, diagonal stripe backgrounds, fixed-width character grids

**References** (from Dribbble collection):
- **ASCII/Terminal patterns**: Blaster UI Aesthetic - bold ASCII art and code-based patterns
- **Brutalist minimalism**: Ali Zafar Iqbal's work (Synthesis, Segments, Tuning In) - clean brutalist layouts with purposeful motion
- **Swiss design execution**: Geometric patterns, systematic color usage demonstrations
- **Motion with purpose**: Obys agency work - sophisticated animations that enhance rather than distract

**Theme**: System-determined (respects user preference), with both light and dark themes executed with equal care. Dark mode with green terminal aesthetics is the default for unauthenticated users, but follows system preference when available.

**Anti-References**:
- Modern SaaS aesthetic (rounded corners, soft shadows, gradient buttons)
- Generic portfolio templates
- Overly decorative or trendy design patterns
- Heavy JavaScript frameworks that sacrifice performance

### Design Principles

1. **Systematic Over Arbitrary**: Every design decision flows from the token system. Spacing, colors, typography all reference predefined foundation tokens that map to semantic meanings.

2. **Performance as Design Quality**: Fast load times, minimal JavaScript, optimized fonts, and efficient rendering aren't just technical requirements - they're core to the brand promise. A slow, bloated site contradicts "performant."

3. **Constraints Breed Creativity**: The limited type scale (4 sizes), zero border radius, and strict spacing system aren't limitations - they're the creative framework. Work within the system to create visual interest through rhythm, hierarchy, and intentional breaks.

4. **Terminal Nostalgia with Modern Execution**: Embrace the retro computing aesthetic through monospace fonts, green accents, and command-line patterns, but implement with modern CSS (OKLCH colors, container queries, cascade layers) for a forward-looking technical foundation.

5. **Quiet Confidence**: Let the systematic approach and attention to detail speak for itself. No need for aggressive CTAs, marketing copy, or visual flourishes. The work demonstrates the capability.

### Technical Constraints

**Framework**: Astro static site with Web Components
**Browser targets**: Modern evergreen browsers (no IE11, leverage modern CSS)
**Performance budget**: 
- First contentful paint < 1s
- Total blocking time < 200ms
- Minimal JavaScript (leverage web components for interactivity)

**Accessibility**: WCAG 2.1 AA minimum, with careful attention to:
- Color contrast (OKLCH system ensures perceptually uniform colors)
- Keyboard navigation (all components support form participation)
- Reduced motion (respect `prefers-reduced-motion`)

### Color System

**Seed Colors**:
- Primary: `#10b981` (terminal green)
- Neutral: `#8b8c93` (tinted gray)
- Constructive: `#5dbb55` (success green)
- Destructive: `#ef5655` (error red)

**Color Philosophy**:
- Generated programmatically using `color-mix()` in OKLAB
- Each seed expands to 11-step scale (50-950)
- Semantic tokens (softer, soft, mid, loud, louder) map to scale values
- Theme switching inverts the mappings while maintaining contrast
- Neutrals tinted toward brand hue for subtle cohesion

### Typography

**Font Families**:
- **Body/UI**: IBM Plex Mono (400, 500, 700) - Clean, readable monospace
- **Display**: Departure Mono (400, 700) - Retro bitmap aesthetic

**Type Scale** (UI-focused, fixed rem):
- `--h1`: 2rem bold (Departure Mono) - Hero text, page titles
- `--h2`: 1.5rem medium (IBM Plex Mono) - Section headings
- `--p`: 1rem regular (IBM Plex Mono) - Body text
- `--p2`: 0.875rem regular (IBM Plex Mono) - Captions, metadata

**Philosophy**: Limited scale creates hierarchy through weight and color, not size variation. Monospace throughout reinforces terminal aesthetic and creates natural grid alignment.

### Spacing System

**Foundation Scale**: 4pt base unit
- Fixed: `--size-000` through `--size-15` (-.5rem to 30rem)
- Fluid: `--size-fluid-1` through `--size-fluid-10` (responsive using clamp)
- Content: `--size-content-1` through `--size-content-3` (character-based widths)

**Semantic Tokens**:
- `--body-margin`: var(--size-fluid-2) - Standard page inset
- `--menu-height`: var(--size-6) - Fixed navigation height

**Philosophy**: Constrained values speed decisions and ensure consistency. Fluid scales breathe on larger viewports while maintaining minimum/maximum bounds.

### Layout Philosophy

**Primitives Over Utilities**:
- Grid, Stack, Collection, Content Grid as building blocks
- Data attributes for configuration (`data-cols`, `data-gap`, `data-padding`)
- Exposed grid on homepage creates intentional asymmetry

**Responsive Approach**:
- Fluid spacing and typography scales reduce breakpoint needs
- Container queries for component-level responsiveness
- Mobile-first with progressive enhancement

### Component Philosophy

**Web Components**:
- Custom elements with form participation (`m-input`, `m-listbox`, `m-textarea`)
- Slots for composition
- Minimal JavaScript, maximum browser platform usage

**Interaction States**:
- Zero border radius maintained in all states
- Subtle color shifts using semantic tokens
- No animations by default (respect reduced motion)
- Focus indicators using `--outline` system

### Motion (When Used)

**Principles**:
- Purposeful, not decorative
- Respect `prefers-reduced-motion`
- Fast, crisp transitions (< 200ms for micro-interactions)
- Use transform and opacity only (avoid layout thrashing)

**Easing**: Linear or subtle ease-out (no bounce, elastic)

### Documentation as Design

The design system documentation IS part of the product. It demonstrates:
- Systematic thinking through comprehensive token reference
- Technical depth through code examples
- Accessibility through semantic HTML and ARIA
- Swiss design through tabular layouts and precise alignment

Every documentation page reinforces the brand through its execution.
